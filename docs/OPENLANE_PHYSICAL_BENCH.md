# OpenLanePhysicalBench — ASIC Physical-Flow Readiness Benchmark

## Overview

OpenLanePhysicalBench checks whether the tiny public-safe DTL gate design
generated by TinyTapeoutPrep can survive a reproducible ASIC physical-flow
readiness pipeline. It extends Phase 9 (TinyTapeoutPrep) with additional
stages that validate OpenLane/OpenROAD toolchain availability, config
correctness, report parsing, and GDS artifact hashing.

## Connection to TinyTapeoutPrep (Phase 9)

OpenLanePhysicalBench builds on the artifacts produced by TinyTapeoutPrep:

- The core RTL, TinyTapeout wrapper, pinout map, and config files from Phase 9
  are used as inputs to the physical flow.
- TinyTapeoutPrep generates structurally valid TinyTapeout-compatible artifacts.
  OpenLanePhysicalBench checks whether those artifacts can survive an
  additional ASIC physical-flow readiness pipeline.
- Phase 9 handles generation and structural validation; Phase 10 adds
  physical-flow config validation, report parsing, and GDS hashing.

See [TINYTAPEOUT_PREP.md](TINYTAPEOUT_PREP.md) for Phase 9 details.

## Public Disclaimer

> OpenLanePhysicalBench checks whether a tiny DTL gate design can move from open-silicon preparation toward a reproducible ASIC physical-flow readiness review. It does not guarantee silicon correctness, fabrication readiness, timing signoff, real power, real area, physical durability, regulatory conformance or safety-critical deployment.

## Usage

```bash
# Run demo with built-in designs and fixtures
python -m chipgate physical --demo

# Run with JSON output from benchmark directory
python -m chipgate physical benchmarks/openlanephysical_v0 --json

# Parse existing OpenROAD/OpenLane reports from fixtures
python -m chipgate physical --parse-reports benchmarks/openlanephysical_v0/fixtures

# Check toolchain availability (OpenLane, OpenROAD, Yosys, Magic, etc.)
python -m chipgate physical --toolchain-status

# Generate HTML report
python -m chipgate physical --demo --html physical_report.html

# Allow unsafe designs to proceed (for testing only)
python -m chipgate physical --demo --allow-unsafe
```

## CLI Reference

```
python -m chipgate physical [--demo] [--json] [--html FILE] [--toolchain-status] [--parse-reports DIR] [--allow-unsafe] [path]
```

| Flag | Description |
|------|-------------|
| `--demo` | Run with built-in demo designs and fixtures |
| `--json` | Output results as JSON to stdout |
| `--html FILE` | Generate self-contained HTML report to FILE |
| `--toolchain-status` | Show OpenLane/OpenROAD toolchain availability and exit |
| `--parse-reports DIR` | Parse OpenROAD report fixtures from DIR and exit |
| `--allow-unsafe` | Allow unsafe designs to proceed past preflight |
| `path` | Path to benchmark directory (default: built-in demo) |

## Stages

OpenLanePhysicalBench runs a 7-stage pipeline for each design:

### Stage 1: Preflight Safety

- Runs the existing ChipGate safety scan (CG001–CG014) on the RTL.
- Checks TinyTapeoutPrep readiness (wrapper, pinout, docs).
- Detects private/proprietary name leaks.
- Unsafe designs are blocked unless `--allow-unsafe` is passed.

### Stage 2: Toolchain Status

Checks availability of the OpenLane/OpenROAD toolchain:

| Tool | Binary | Role |
|------|--------|------|
| OpenLane | `openlane` | Full ASIC flow orchestrator |
| OpenROAD | `openroad` | Physical synthesis, P&R, STA |
| Yosys | `yosys` | Logic synthesis |
| Magic | `magic` | Layout editor, GDS streaming |
| Netgen | `netgen` | Netlist comparison (LVS) |
| KLayout | `klayout` | Layout viewer, GDS validation |
| STA | `sta` / `opensta` | Static timing analysis |

Tools are optional. Missing tools produce `*_SKIPPED_TOOL_MISSING` statuses.

### Stage 3: Config Validation

Validates the OpenLane configuration file for the design. Checks required
fields, PDK references, clock definitions, and structural correctness.

### Stage 4: Optional OpenROAD/OpenLane Run

If OpenLane or OpenROAD is installed, attempts to run the physical flow.
Gracefully skipped if tools are missing. Results depend on tool versions and
configuration.

### Stage 5: Report Parsing

Parses OpenROAD/OpenLane reports from fixtures:

- **DRC reports** — Design rule check violation counts and summaries.
- **LVS reports** — Layout vs. schematic mismatch detection.
- **Timing reports** — Worst negative slack (WNS), setup/hold checks.
- **Area stats** — Die area, utilisation, cell counts.

When no official flow was run, fixture reports provide parsed data for
structural validation.

### Stage 6: GDS Artifact Hashing

If a GDS file exists, it is SHA-256 hashed. If no GDS file exists,
all text artifacts (RTL, wrapper, config, pinout, reports, replay command)
are hashed instead. See [GDS_ARTIFACTS.md](GDS_ARTIFACTS.md) for details.

### Stage 7: Evidence Pack

Generates a reproducible evidence record with all artifact hashes, statuses,
metrics, manual review items, public wording, and limitation text.

## Status Constants (Phase 10)

OpenLanePhysicalBench introduces 21 new status constants:

| # | Status | Meaning |
|---|--------|---------|
| 1 | `PHYSICAL_BENCH_PASS` | All designs passed the physical flow pipeline |
| 2 | `PHYSICAL_BENCH_FAIL` | One or more designs failed a physical flow stage |
| 3 | `PHYSICAL_FLOW_READY` | Physical flow readiness checks passed |
| 4 | `PHYSICAL_FLOW_SKIPPED_TOOL_MISSING` | Physical flow skipped — toolchain not installed |
| 5 | `OPENLANE_CONFIG_PASS` | OpenLane configuration file is valid |
| 6 | `OPENLANE_CONFIG_FAIL` | OpenLane configuration file has errors |
| 7 | `OPENROAD_RUN_PASS` | OpenROAD physical flow run succeeded |
| 8 | `OPENROAD_RUN_FAIL` | OpenROAD physical flow run failed |
| 9 | `OPENROAD_SKIPPED_TOOL_MISSING` | OpenROAD skipped — not installed |
| 10 | `DRC_CLEAN` | DRC report parsed with zero violations |
| 11 | `DRC_VIOLATIONS_FOUND` | DRC report parsed with one or more violations |
| 12 | `DRC_SKIPPED_NO_REPORT` | DRC check skipped — no report file found |
| 13 | `LVS_CLEAN` | LVS report parsed with zero mismatches |
| 14 | `LVS_MISMATCH_FOUND` | LVS report parsed with one or more mismatches |
| 15 | `LVS_SKIPPED_NO_REPORT` | LVS check skipped — no report file found |
| 16 | `TIMING_REPORT_PASS` | Timing report parsed with no negative slack |
| 17 | `TIMING_REPORT_FAIL` | Timing report parsed with negative slack |
| 18 | `TIMING_REPORT_SKIPPED` | Timing check skipped — no report file found |
| 19 | `GDS_HASH_CREATED` | GDS file found and SHA-256 hash computed |
| 20 | `GDS_MISSING` | No GDS file available for hashing |
| 21 | `NEEDS_OFFICIAL_OPENLANE_RUN` | Results are from fixtures — official flow required |

See `chipgate/statuses.py` for the full list including statuses shared
with other benchmarks (`NEEDS_PHYSICAL_SIGNOFF`, `NEEDS_MANUAL_REVIEW`).

## Metrics

| Metric | Description |
|--------|-------------|
| `designs_tested` | Number of designs processed |
| `physical_configs_checked` | Number of OpenLane config validations |
| `openlane_config_pass_rate` | Fraction of configs passing validation |
| `openroad_run_pass_rate` | Fraction of OpenROAD runs succeeding |
| `drc_violation_count` | Total DRC violations across all designs |
| `lvs_mismatch_count` | Total LVS mismatches across all designs |
| `worst_negative_slack` | Worst negative slack (ns) across timing reports |
| `gds_artifact_count` | Number of designs with a GDS file |
| `artifact_hash_count` | Total SHA-256 artifact hashes |
| `replay_match_rate` | Deterministic replay consistency |
| `evidence_packs_created` | Number of evidence records created |
| `toolchain_coverage` | Fraction of tools found vs checked |
| `manual_review_items` | Count of items requiring manual review |

## What the Benchmark Does NOT Prove

OpenLanePhysicalBench does **not** prove or guarantee:

- **Silicon correctness** — No timing analysis signoff, no DRC/LVS foundry
  signoff, no silicon debug or validation.
- **Fabrication readiness** — No GDS production, no foundry PDK qualification,
  no manufacturing test coverage.
- **Timing closure** — Parsed timing reports are from fixtures unless an
  official flow was run. No STA signoff.
- **Power characterisation** — No dynamic or leakage power measurement. No
  thermal analysis.
- **Real area** — Area stats are from fixture reports. No gate-level area
  measurement after synthesis.
- **Safety certification** — No IEC 61508, ISO 26262, DO-254, or medical
  device certification.
- **Tiny Tapeout acceptance** — No official CI integration, no GDS build
  verification by tinytapeout.com.
- **NVIDIA comparison** — No comparison to any commercial or proprietary
  chip design tool or workflow.
- **Medical, defence, or robotics use** — Not certified or validated for
  safety-critical deployment in any regulated domain.

## Graceful Degradation

All external tools (OpenLane, OpenROAD, Yosys, Magic, Netgen, KLayout, STA)
are optional. Missing tools produce `*_SKIPPED_TOOL_MISSING` statuses. No
external tool is required for demo mode — fixture data is used instead.

## Module Structure

| Module | Role |
|--------|------|
| `openlane_physical.py` | Main orchestrator, design processing, toolchain checks |
| `openroad_reports.py` | OpenROAD fixture report parsing |
| `drc_lvs_parser.py` | DRC and LVS report parsing |
| `timing_report_parser.py` | Timing report and area stats parsing |
| `gds_artifacts.py` | GDS file hashing and artifact hash tables |
| `physical_score.py` | Aggregate metrics and scoring |
| `physical_report.py` | Static HTML report generation |

## Output Files

- **JSON**: Full structured output with all design results, toolchain report,
  metrics, and evidence records.
- **HTML**: Self-contained report with summary cards, design table, toolchain
  status, metrics panel, evidence hashes, manual review items, and limitations.
- **Evidence pack**: `physical_evidence.json` with SHA-256 hashes, statuses,
  public wording, and limitation text.
